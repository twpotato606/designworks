上線後陸續發現的問題

主設計師離職，設計文件只有操作方式，業務邏輯幾乎沒有記錄
系統帶著一堆未發現的 bug 就直接上線了
設計到工程的溝通斷層，在業務邏輯尚未釐清的狀態下，工程師自行決定怎麼實作

短期解決方案

跳下去兼職客服，親自記錄遇到問題與背後脈絡
短期無法修的 bug，幫忙想如何繞路先讓使用者可以繼續操作
推動新機制：所有業務邏輯全團隊與客戶釐清後才實作，並記錄到設計文件中

"設計師不是把設計稿交付就結束了，而是要確保系統能真的順暢地運作起來。"

Wireflow Viewer  —  工具說明

當 Vibe Coding 夠快也夠精準時，還需要手繪設計稿嗎？

速度
生成速度遠超手繪
有人說 Vibe 風格會跑掉——但套用既有設計系統元件，輸出的介面感覺幾乎一致。
精準度
看得懂 CSS 就能微調
難以微調是對一般人而言——精確的指令下達，不是口語模糊地溝通，細節完全可控。
唯一缺口
雛形無法取代文件
不能只交付雛形——功能定義、流程、例外狀況，stakeholder 和開發仍需要一份可查看的文件。

自產工具迭代歷程

v1
雛形 → JSON → Figma
瀏覽器插件爬蟲 + Figma 插件轉換，但倒入後即脫離 AI 維護能力
v2
雛形 → JSON → VS Code Plugin
終點改為自製 VS Code 插件，JSON 原始檔可由 AI Agent 直接改動
v3
Wireflow Viewer — 今天 Demo 的工具
真實元件 · 無限畫布 · AI 自動擷取，完整設計流程文件一鍵生成
Wireflow Viewer  —  工具說明 · v1

v1 — 自動爬蟲雛形，
直接倒回 Figma

架構流程

1
設計師 vibe coding 產出雛形
2
AI 分析原始碼，產生流程擷取計畫
3
自製瀏覽器插件，依計畫爬蟲取得 JSON
4
自製 Figma 插件，將 JSON 轉換倒回 Figma
缺點
倒入 Figma 後脫離 AI 維護能力
一旦設計稿在 Figma 中，就回到了手動建元件、維護元件庫的老路——Vibe Coding 的快速迭代優勢消失了。

Ch.4  —  對未來設計的想像與期許

回頭看，
被退學也許不是壞事

找到熱情與長處的交集
設計 × 使用者研究 × Coding
如果不是被退學，不會花三個月沉思，不會轉到資訊傳播，不會選設計運算所——那段彎路，是我現在能力組合的起點。
在 DeepQ 的體會
懂技術的設計師，在這個時代有更多的優勢
當 AI 工具讓 coding 門檻降低，能快速驗證想法、自己做出工具的設計師，在設計上更有發揮的空間。
我的看法

在未來幾年內，AI 的能力只會越來越強。
設計師比拚的，不再是誰的介面畫得更漂亮、更精細。

而是誰的迭代速度更快，精準抓到使用者的需求與痛點並解決它。 而這正是我的優勢——
