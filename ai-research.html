傳統設計日常：高昂的手作成本

核心觀念：「線性增長的工時」——畫面數量翻倍，工時同步翻倍，設計師淪為工廠裡的零件組裝工，而非創造價值的體驗設計者。

手工成本

每一像素、每一份圖文說明皆需手寫。100 個畫面即 100 倍工時，無法規模化複製或快速迭代。

Design System 隱形負擔

Design System 雖減少重複製圖，卻大幅增加維護組件的隱形成本與元件如何使用的溝通摩擦。

根本問題：價值錯置

設計師精力耗費在組裝零件，而非聚焦於真正創造使用者價值的體驗決策與策略思考。

典範轉移

2026 核心願景：Vibe Coding to Design

核心觀念：「從個別畫面轉向意圖生成」——不再逐頁繪製，而是透過 AI 生成產品全流程雛形，在進入開發前完成設計驗證。

實作流程

1

建構設計系統環境

採用現成設計系統或自製 .tsx 元件庫，確立視覺語言基礎

2

進行雛形開發

透過 AI 輔助快速生成產品流程雛形，取代逐頁繪製

3

實際體驗感受並"即時"修正

在可互動的真實環境中驗證體驗，而非靜態 mockup

4

使用 Wire-flow Viewer擷取雛形操作流程

自動生成雛形中的使用者流程與相關畫面的規格說明

5

在Viewer 中查看產品操作流程文件

擷取後透過 viewer 確認完整產品設計流程文件

6

設計流程文件交付開發

附帶完整流程文件的雛形直接移交工程師，無需額外補齊規格

這麼做的好處

設計師從執行者轉型為產品體驗的「導演」，專注策略與創意決策，由 AI 負責執行層面的批次生成

AI 直接從修正後的 code 反向更新規格文件，始終與實作同步，避免浪費時間在對齊文件規格上

雛形與規格相輔相成，PM、設計師、工程師在同一個 Git 倉庫持續迭代——版控即協作，設計交付不再是單向移交

關鍵挑戰：Vibe Coding雛形 ≠ 可交付開發

溝通鴻溝：單純的 Prototype 缺乏開發所需的深度資訊。工程師需要包含架構說明、例外處理與資料邏輯的圖文流程文件。若手動補齊文件，AI 省下的效率將被完全抵消，形成新的浪費循環。

工程師實際需要的圖文流程文件，涵蓋架構說明、例外狀態與資料邏輯，遠超一般原型所能提供的資訊深度。

Wireflow Viewer：直接將雛形轉成設計流程文件

無需爬蟲，直接 import 原專案的真實 React .tsx元件，以 iframe 節點形式渲染在無限畫布上，搭配 CaptureFlow AI 腳本，讓 AI 自動分析原始碼並產出完整 Wireflow。

Figma 的限制
  • 無法直接查詢產品流程規格
  • 需手動更新元件/介面最新樣式
  • 不支援設計修改同步至 Vibe Coding 專案(2026 Figma mcp已經可以穩定支援)
Wireflow Viewer 的能力
  • 可透過AI查詢流程完整規格文件
  • 可自動更新介面樣式
  • 一鍵同步設計修改至 Vibe Coding 專案

Wireflow Viewer：工具核心能力

一套基於 React + Vite 的互動式流程畫布工具，將成熟(或自創)的設計系統 .tsx元件以「畫面節點」形式渲染在 iframe 中，並以有向箭頭連接，可視化呈現完整的使用者操作流程。

直接 import 原始元件

Wrapper 透過引用原專案的頁面與 Layout，設計文件上畫面 100% 與雛形一致，原始碼更新後 Wireflow 自動同步。不複製程式碼，不脫離開發環境。

AI 自動擷取（CaptureFlow）

只需對 AI 說出指令並附上腳本，AI 自動分析 React 原始碼、識別所有畫面狀態與轉換觸發條件，自動產出 Wrapper.tsx 與 Wireflow JSON。

版控整合 Git

PRD、Wireflow、Prototype 全納入 Git，歷史可追溯、變更可回滾。PM、設計師、工程師不需要切換工具，可隨時取用所需的資料。

三角色無縫協作

PM、設計師、工程師同處一個環境，消除工具切換與資訊落差。修改後直接發起討論，指定 PIC Review，持續推進專案進度不中斷。
